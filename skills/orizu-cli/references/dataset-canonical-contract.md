# Dataset Canonical Contract (Distributed Skill Copy)

This is the portable, skills-facing copy of the Dataset contract for Orizu CLI workflows.
It mirrors the canonical internal contract used in the source repository.

## Canonical Dataset Object

Required metadata contract:

- `id`: immutable dataset id (UUID string)
- `project_id`: owning project id (UUID string)
- `name`: human-readable dataset name
- `row_count`: externally visible row count
- `created_at`, `updated_at`: timestamps

Optional metadata:

- `source_type`
- `source_config` (may include object-store pointers such as `object_store.bucket`, `object_store.path`, `object_store.format`)
- `source_message_id`
- `source_attachment_name`
- `created_by`

## Canonical External Row Identity

- `row.id` is required and is the canonical row identifier.
- Row payload is the rest of the JSON object.
- `row_index` is not canonical identity.

Compatibility rule:

- Canonical API/CLI/UI runtime paths use row-id selectors only.
- `row_index` is not part of canonical runtime selection contracts.

## Ordering and Count Semantics

- Ordering must be deterministic for a given snapshot.
- Ordering never defines row identity.
- `row_count` is computed at read time (not persisted as a physical `datasets.row_count` column).
- Runtime count computation uses canonical rows with optional object-store metadata fallback.

## Lineage and Lock Semantics

- Derived datasets are independent copies, not live pointers.
- Lineage metadata is explicit (commonly via source metadata fields).
- Lock is one-way; locked datasets reject mutation operations.
- Lock state is persisted via `datasets.locked_at` and `datasets.locked_by`.
- Canonical row hard-deletes must fail when targeted rows are assignment-referenced.

## Hugging Face Compatibility Profile

Row mapping:

- HF rows map to Orizu row objects.
- Missing `id` values are generated by Orizu.

Features/schema expectations:

- Preserve column names and JSON-compatible value types.
- Record any lossy coercions in source metadata.

Split handling:

- Preserve split metadata (for example `train`, `validation`, `test`) in source metadata.

Public/private access and auth:

- Preserve source visibility metadata (`public`, `private`, or `gated`) when known.
- Public HF datasets should not require auth material.
- Private/gated HF datasets require runtime auth, but raw tokens must never be persisted in row payloads, dataset metadata, exports, or logs.
- Store only non-secret credential references in source metadata (for example `huggingface.token_ref` or integration identifiers).

Metadata mapping:

- Preserve upstream metadata (description, license, config/revision/source references) in source metadata.

Practical interchange:

- Current CLI/runtime interchange remains `csv`, `json`, `jsonl`.
- HF compatibility is enforced as row-object plus metadata compatibility.
- Preview/export should preserve one canonical row envelope for db-backed and object-store-backed dataset sources.
